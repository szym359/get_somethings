# Skill 是什么

skill 可以理解成：

> 放在项目或用户目录里的、可被 Codex 按名称或场景调用的一份专用说明书。

它适合保存一类重复使用的工作方式，例如：

- 客服回复规则
- 小红书文案风格
- 代码审查流程
- 法律风险检查清单
- 固定格式的报告生成流程
- 需要配合模板、脚本、素材使用的任务

## Skill 解决什么问题

如果你每次都要在聊天里重复说一大段规则，比如：

```text
回复客户时语气要克制，先承认问题，再给方案，不要承诺超出政策的补偿。
```

那么可以把这段规则沉淀成 skill。以后你只要说：

```text
用 $customer-reply 回复这段投诉
```

Codex 就会先读取这个 skill，再按里面的规则处理。

## Skill 和普通提示词的区别

普通提示词只在当前对话里生效。

skill 是结构化保存下来的规则，可以被反复调用，也可以配套引用资料、模板、脚本和示例。

## Skill 的最小组成

一个最小 skill 只需要一个文件：

```text
.agents/skills/my-skill/SKILL.md
```

内容类似：

```markdown
---
name: my-skill
description: 仅当用户明确要求使用 my-skill 时使用。
---

# My Skill

- 规则 1
- 规则 2
- 规则 3
```

其中：

- `name` 是 skill 的真实名称，触发时通常用 `$name`
- `description` 描述什么时候应该使用这个 skill
- 正文写具体工作流程、规则和注意事项

## 什么时候适合做成 skill

适合：

- 同一类任务会反复出现
- 规则比较稳定
- 规则超过几句话，容易忘或容易说漏
- 需要引用案例、模板、脚本、品牌规范或业务文档
- 你希望不同场景 A/B/C/D 有不同处理方式

不适合：

- 只用一次的临时要求
- 还没有稳定下来的想法
- 项目所有任务都必须遵守的默认规则

最后一种更适合放到 `AGENTS.md`。
